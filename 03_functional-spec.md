## 공통 규칙

### 공통 응답 포맷
```json
// 성공
{ "success": true, "data": { ... }, "meta": { "page": 1, "total": 100 } }
// 실패
{ "success": false, "error": { "code": "AUTH_001", "message": "..." } }
```

### 공통 에러 코드
```
AUTH_001: 인증 실패 / AUTH_002: 토큰 만료 / AUTH_003: 권한 없음
VALID_001: 필수 파라미터 누락 / VALID_002: 파라미터 형식 오류 / VALID_003: 값 범위 오류
RES_001: 리소스 없음(404) / SERVER_001: 서버 오류(500)
RENT_001: 이미 납부 처리된 건 / SETTLE_001: 정산 대상 계약 아님
```

## FR-AUTH-001 간편인증 로그인

**관련 화면**: COM-003 | **우선순위**: [P1] | **관련 SRS**: FR-AUTH-001

### 개요
> 휴대폰 본인인증(PASS 또는 카카오)만으로 회원가입과 로그인을 동시에 처리한다. 별도 ID/PW 생성 절차는 없다.

### 트리거
- COM-002에서 역할(임대인/임차인) 선택 후 인증수단 버튼 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| authProvider | string | Y | "pass" \| "kakao" |
| authToken | string | Y | 인증사 발급 토큰 |
| role | string | Y | "landlord" \| "tenant" |

### 처리 로직
1. 클라이언트: 인증수단 버튼 탭 → 해당 SDK 인증창 호출
2. 인증 완료 시 authToken 수신 → 서버로 전달
3. 서버: authToken 검증 → 최초 사용자면 자동 가입 처리, 기존 사용자면 로그인 처리
4. 서버: JWT Access Token(24h) + Refresh Token(30d) 발급
5. 클라이언트: 토큰 저장 후 role에 따라 LL-001(임대인) 또는 TN-001(임차인, 계약연결 필요 시 COM-007 경유)로 이동
6. 최초 가입자는 COM-004(1분 안내)로 우선 이동

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 인증 실패/취소 | AUTH_001 | "인증에 실패했습니다. 다시 시도해주세요." | 인증 버튼 화면 복귀 |
| 네트워크 오류 | SERVER_001 | "네트워크 오류가 발생했습니다." | 토스트 + 재시도 버튼 |
| 토큰 만료 | AUTH_002 | (자동 재인증 유도) | 재로그인 화면 이동 |

### API 엔드포인트
```
POST /api/v1/auth/login
Request:  { "authProvider": "kakao", "authToken": "...", "role": "landlord" }
Response: { "success": true, "data": { "accessToken": "...", "refreshToken": "...", "user": { "id": 1, "role": "landlord" }, "isFirstLogin": true } }
```

## FR-PROPERTY-001~002 건물 등록 및 호실 등록

**관련 화면**: LL-002, LL-002a, LL-002b | **우선순위**: [P1] | **관련 SRS**: FR-PROPERTY-001~005

### 개요
> 임대인은 건물을 먼저 등록하고, 건물 하위에 호실을 개별 등록한다. 호실은 면적/구조/옵션/특기사항을 카드 선택 중심으로 입력받는다.

### 트리거
- LL-001에서 [+ 물건 추가] 탭 → 건물 등록 → 완료 시 LL-002a(호실목록)에서 [+ 호실 추가] 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| buildingAddress | string | Y | 주소 검색 API 결과값 |
| buildingType | string | Y | "원룸" \| "다가구" \| "오피스텔" \| "상가주택" 등 사전정의 |
| totalUnitCount | number | Y | 1 이상 |
| unitNumber | string | Y | 예: "101호" |
| floor | number | N | 정수 |
| exclusiveArea | number | Y | 제곱미터, 0 초과, 숫자패드 |
| structureType | string | Y | "원룸" \| "투룸" \| "쓰리룸이상" 카드선택 |
| options | string[] | N | 다중선택 카드("에어컨","냉장고","세탁기","붙박이장","가스레인지","엘리베이터" 등) |
| specialNotes | string | N | 프리셋 우선, "직접입력" 버튼 클릭 시에만 텍스트 노출 |

### 처리 로직
1. 건물 등록: 주소 검색 → 건물유형 카드선택 → 총 호실수 숫자패드 → 건물 외관사진 촬영
2. 등록 완료 시 총 호실수만큼 "호실번호 미입력" 빈 슬롯이 LL-002a에 자동 생성(예: 101호~ 형태 자동 제안, 수정 가능)
3. 각 호실 슬롯 탭 → LL-002b 진입 → 면적(숫자패드) → 구조(카드) → 옵션(다중선택 카드, 최대 개수 제한 없음) → 특기사항(프리셋 카드, 필요 시 직접입력)
4. 저장 시 호실 상태를 "공실"로 초기화, LL-002a 목록에 즉시 반영
5. 계약 등록(LL-003) 완료 시 호실 상태가 "임대중"으로 자동 전환

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 면적 0 또는 음수 | VALID_003 | "면적을 다시 확인해주세요." | 인라인 에러 |
| 총 호실수보다 많은 호실 등록 시도 | VALID_003 | "등록 가능한 호실 수를 초과했습니다." | 총 호실수 수정 안내 |
| 주소 검색 결과 없음 | RES_001 | "주소를 찾을 수 없습니다. 도로명으로 다시 검색해주세요." | 재검색 유도 |

### API 엔드포인트
```
POST /api/v1/buildings
Request:  { "address": "서울시 강남구...", "buildingType": "다가구", "totalUnitCount": 6 }
Response: { "success": true, "data": { "buildingId": "bld_1", "unitSlots": ["101","102","201","202","301","302"] } }

POST /api/v1/buildings/{buildingId}/units
Request:  { "unitNumber": "101호", "exclusiveArea": 23.5, "structureType": "원룸", "options": ["에어컨","냉장고"], "specialNotes": null }
Response: { "success": true, "data": { "unitId": "unit_1", "status": "공실" } }
```

## FR-PROPERTY-008 임차인 변경

**관련 화면**: LL-002c, LL-003a | **우선순위**: [P1] | **관련 SRS**: FR-PROPERTY-008, FR-PROPERTY-009

### 개요
> 기존 임차인의 계약 종료(퇴거정산 자동 연계)와 신규 임차인 계약 등록을 하나의 흐름으로 이어 붙여, 임대인이 두 번 일하지 않도록 한다.

### 트리거
- LL-002c 호실상세에서 [임차인 변경] 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| unitId | string | Y | 존재하는 호실 ID, 상태 "임대중"이어야 함 |
| confirmSettlementDone | boolean | Y | 퇴거정산(FR-SETTLE) 완료 여부 |

### 처리 로직
1. [임차인 변경] 탭 → 기존 계약이 "정산 완료" 상태인지 확인
2. 미완료 시: LL-013(퇴거 정산 계산기)로 강제 이동, 정산 서명 완료 전에는 다음 단계 진행 불가
3. 정산 완료 확인 시: 기존 계약 상태를 "종료"로 변경, 임차인 정보는 호실 이력(FR-PROPERTY-009)으로 이관되어 보존
4. 호실 상태를 "공실"로 전환
5. 바로 이어서 LL-003(계약 등록) 화면으로 이동해 신규 임차인 정보 입력 진행(입주키트는 기존 값이 프리필되어 표시 — 임대인이 "이전과 동일" 선택 시 재입력 없이 재사용 가능)
6. 신규 계약 등록 및 초대링크 발급 완료 시 호실 상태 "임대중"으로 전환

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 정산 미완료 상태에서 변경 시도 | SETTLE_001 | "퇴거 정산을 먼저 완료해주세요." | 정산 화면으로 강제 이동 |
| 공실 상태 호실에서 변경 시도 | VALID_003 | "현재 공실 상태입니다. 신규 계약 등록을 진행해주세요." | LL-003으로 바로 이동 |
| 입주키트 프리필 값 임차인 확인 전 노출 오류 | SERVER_001 | "정보를 불러오지 못했습니다." | 새로고침 버튼 |

### API 엔드포인트
```
POST /api/v1/units/{unitId}/tenant-change
Request:  { "confirmSettlementDone": true }
Response: { "success": true, "data": { "previousContractId": "ctr_1", "unitStatus": "공실", "prefillKitAvailable": true } }
```

### Use Case 시나리오
**정상**: 정산 완료된 호실에서 [임차인 변경] → 자동 공실 전환 → 신규 계약 등록(입주키트 "이전과 동일" 선택) → 초대링크 발급 → 임대중 전환
**예외**: 정산 미완료 상태에서 변경 시도 → 정산 화면으로 강제 이동 → 정산 완료 후 재시도

## FR-PROPERTY-010~012 공실 현황 대시보드 및 계약갱신 알림

**관련 화면**: LL-022, LL-023 | **우선순위**: [P1] | **관련 SRS**: FR-PROPERTY-010~012

### 개요
> 건물별 공실률과 호실별 공실기간을 자동 집계하고, 계약만료 임박 호실을 D-60/D-30 기준으로 자동 알림한다.

### 트리거
- LL-001 하단 탭에서 진입, 또는 매일 배치 스케줄러가 계약만료일 대비 경과일 확인

### 처리 로직
1. **공실 현황 집계**: 호실 상태가 "공실"인 기간을 공실 전환 시점부터 실시간 계산, 건물 단위로 공실률(%) = 공실 호실수 / 총 호실수 자동 산출
2. LL-022: 건물별 카드에 공실률 %와 공실 호실 목록(공실 기간 순 정렬) 표시
3. **계약갱신 알림**: 매일 배치가 전체 계약의 종료일 대비 D-60, D-30 시점을 확인해 임대인에게 알림 발송(FR-NOTIFY-001 재사용)
4. LL-023: 계약만료 임박 호실을 D-day 뱃지와 함께 목록으로 표시, 목록에서 바로 FR-PROPERTY-008(임차인 변경) 또는 갱신(FR-AMOUNT-001) 플로우로 진입 가능

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 계약 종료일 정보 누락 | VALID_001 | "계약기간 정보가 없어 알림을 보낼 수 없습니다." | 계약상세에서 종료일 입력 유도 |
| 알림 발송 실패 | SERVER_001 | (임대인 화면에 실패 표시) | LL-023에 재발송 버튼 |

### API 엔드포인트
```
GET /api/v1/buildings/{buildingId}/vacancy-summary
Response: { "success": true, "data": { "vacancyRate": 16.7, "vacantUnits": [{ "unitId": "unit_3", "vacantDays": 42 }] } }

GET /api/v1/contracts/renewal-alerts?dueWithinDays=60
Response: { "success": true, "data": [{ "contractId": "ctr_5", "unitNumber": "202호", "daysRemaining": 28 }] }
```

**관련 화면**: LL-004 | **우선순위**: [P1] | **관련 SRS**: FR-KIT-001

### 개요
> 공동현관/세대현관/우편함/주차 비밀번호를 숫자패드로 입력받아 암호화 저장한다. 문자 키보드는 노출하지 않는다.

### 트리거
- LL-003 계약등록 완료 후 자동 진입, 또는 LL-016 계약상세에서 수정 진입

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| contractId | string | Y | 존재하는 계약 ID |
| commonDoorPw | string | N | 숫자 4~8자리 |
| unitDoorPw | string | N | 숫자 4~8자리 |
| mailboxPw | string | N | 숫자 2~6자리 |
| parkingInfo | string | N | 숫자 또는 사전 정의 선택값 |

### 처리 로직
1. 화면 진입 시 4개 항목 카드가 순차 노출, 각 카드는 숫자패드 UI만 제공
2. 항목별 "건너뛰기"(추후 입력) 버튼 상시 노출 — 전체 입력 강제하지 않음
3. 입력 완료 항목마다 클라이언트에서 즉시 서버 전송(자동 저장), 화면 이탈해도 유실 없음
4. 서버: AES-256으로 암호화하여 저장
5. 전체(또는 건너뛴 항목 포함) 완료 시 LL-005(설비정보)로 자동 이동
6. 임차인 승인 전까지는 TN-003에서 열람 불가 — 계약 서명 완료(SIGN-002) 후에만 노출

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 형식 오류(자릿수 초과) | VALID_002 | "숫자 4~8자리로 입력해주세요." | 인풋 하단 인라인 표시, 숫자패드 유지 |
| 저장 실패 | SERVER_001 | "저장에 실패했습니다. 다시 시도해주세요." | 토스트 + 재시도, 로컬 임시 보관 후 재전송 |
| 계약ID 없음 | RES_001 | - | 이전 화면으로 강제 이동 |

### API 엔드포인트
```
PATCH /api/v1/contracts/{contractId}/kit/access
Request:  { "commonDoorPw": "1234", "unitDoorPw": "5678" }
Response: { "success": true, "data": { "updatedFields": ["commonDoorPw", "unitDoorPw"] } }
```

### Use Case 시나리오
**정상**: 4개 항목 모두 숫자패드로 입력 → 자동저장 → 다음 화면 이동
**예외(일부 미보유)**: 주차정보 모름 → [건너뛰기] 탭 → 나머지만 저장 → TN-004에서 임차인이 추후 요청 가능

## FR-INVITE-001 초대링크 발급

**관련 화면**: LL-009 | **우선순위**: [P1] | **관련 SRS**: FR-INVITE-001, FR-INVITE-002

### 개요
> 계약 단위로 임차인 초대링크를 생성하고 문자/카카오톡으로 전송한다. 링크는 7일간 유효하다.

### 트리거
- LL-003~LL-008 입주키트 입력 완료(또는 건너뛰기 포함) 후 [초대링크 만들기] 버튼 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| contractId | string | Y | 존재하는 계약 ID |
| tenantPhone | string | Y | 010-XXXX-XXXX 형식 |
| sendChannel | string | Y | "sms" \| "kakao" |

### 처리 로직
1. 임차인 전화번호를 숫자패드로 입력
2. 서버: 계약 연결용 고유 토큰 생성(만료 7일), 단축 URL 발급
3. sendChannel에 따라 SMS 또는 카카오 알림톡으로 발송

## FR-SETTLE-001~004 퇴거 정산 계산

**관련 화면**: LL-013 | **우선순위**: [P1] | **관련 SRS**: FR-SETTLE-001~004

### 개요
> 퇴거일, 검침값, 미납내역을 입력받아 임차료 일할계산·공과금 차감·보증금 반환액을 자동 산출한다.

### 트리거
- LL-010에서 [퇴거정산] 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| contractId | string | Y | 존재하는 계약 ID |
| moveOutDate | date | Y | 계약기간 내 또는 그 이후 날짜 |
| finalGasReading | number | N | 0 이상 |
| finalElectricReading | number | N | 0 이상 |
| unpaidAmount | number | N | 0 이상, 없으면 0 |
| repairDeduction | number | N | 원상복구 협의 완료금액(0 이상) |

### 처리 로직
1. 퇴거일을 캘린더에서 선택
2. 계량기 최종 수치는 사진 촬영 후 OCR 자동인식 또는 숫자패드 직접 입력
3. 미납내역은 LL-019 미납대장에서 자동 연동(수정 가능)
4. 서버: 임차료 일할 정산액 = 월임차료 ÷ 해당월 총일수 × 퇴거월 거주일수
5. 서버: 공과금 차감액 = (최종검침값 - 직전 정산 기준값) × 단가(또는 고지서 기준 직접입력값)
6. 서버: 보증금 반환액 = 보증금 - (미납액 + 공과금차감 + 원상복구협의금) + (선납 임차료 환급분, 있는 경우)
7. LL-013에 산출 내역 항목별 상세 표시, 임대인 확인 후 [정산서 발송]
8. 임차인 TN-007에서 동일 내역 확인 → 양측 서명(SIGN-001) 완료 시 확정, 정산서 PDF 자동 발급(LL-014)

|-----------|-----------|---------------|---------|

## FR-RENT-001~005 임차료 납부 알림/독촉 시퀀스

**관련 화면**: LL-017, LL-018, TN-009 | **우선순위**: [P1] | **관련 SRS**: FR-RENT-001~005

### 개요
> 매월 지정일 기준으로 납부안내→납부확인→미납안내→미납독촉 4단계 알림을 자동 발송하고, 납부상태를 세대별로 관리한다.

### 트리거
- 시스템 스케줄러(매일 00:10 배치)가 각 계약의 납부 기준일 대비 경과일수를 확인

### 처리 로직 (상태 시퀀스)
1. **문자① 납부안내**: 납부기준일 D-3일에 임차인에게 "이번달 임차료 안내" 알림 자동 발송
2. **납부확인**: 임대인이 LL-017에서 입금 확인 후 [납부완료] 수동 처리(2차 범위: 계좌연동 자동확인)
3. **문자② 완료확인** (분기 A): 납부완료 처리 시 임차인에게 완료 확인 알림 자동 발송, 상태 "완료"로 표시
4. **문자③ 미납안내** (분기 B): 납부기준일 경과 후 미확인 상태 D+1일에 자동 발송, 상태 "미납"으로 표시
5. **문자④ 미납독촉**: 미납 상태가 임대인 설정 기준일수(기본 D+7) 이상 지속 시 독촉 알림 자동 발송, 상태 "독촉중"으로 표시
6. 미납 상태 계약은 LL-019 미납자 명단 대장에 자동 집계되며 연체료 자동 계산(FR-RENT-008)이 함께 표시

### 상태값 정의
| 상태 코드 | 의미 |
|-----------|------|
| PENDING | 안내 발송 완료, 납부 대기 |
| PAID | 납부 확인 완료 |
| UNPAID | 기준일 경과, 미확인 |
| OVERDUE | 미납 기준일수 초과, 독촉 단계 |

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 임대인이 이미 완료 처리한 건 재처리 시도 | RENT_001 | "이미 처리된 납부 건입니다." | 목록 새로고침 |
| 알림 발송 실패(수신거부 등) | SERVER_001 | (임대인 화면에 발송실패 표시) | LL-017에 실패 뱃지 노출, 수동 재발송 버튼 |
| 계약 종료 이후 잔여 스케줄 | - | - | 계약 상태 "종료" 시 자동 스케줄 중단 |

### API 엔드포인트
```
GET  /api/v1/contracts/{contractId}/rent/status?month=2026-08
POST /api/v1/contracts/{contractId}/rent/confirm-payment
Request:  { "month": "2026-08", "confirmedBy": "landlord_manual" }
Response: { "success": true, "data": { "status": "PAID", "notifiedAt": "2026-08-05T09:00:00Z" } }
```

### Use Case 시나리오
**정상**: D-3 안내 발송 → 임차인 납부 → 임대인 확인 → 완료 알림 발송 → 상태 PAID
**예외(미납 지속)**: 안내 발송 → 미확인 → D+1 미납안내 → D+7 미납독촉 → 미납자 명단 대장 자동 반영 → 연체료 자동 계산

## FR-RENT-009~010 세무 신고자료 생성 및 대장 내보내기

**관련 화면**: LL-020, LL-019 | **우선순위**: [P2] | **관련 SRS**: FR-RENT-009, FR-RENT-010

### 개요
> 기간을 선택하면 해당 기간의 임대소득 현황(세대별 수령액, 미납액, 연체료 포함)을 종합소득세 신고 참고용 요약 자료로 생성한다. 정식 세무 서식이 아닌 "신고 참고자료"임을 명시한다.

### 트리거
- LL-017에서 [세무 신고자료 생성] 탭

### 입력값
| 파라미터 | 타입 | 필수 | 유효성 검사 |
|----------|------|------|-------------|
| landlordId | string | Y | 로그인 사용자 ID |
| periodStart | date | Y | 연 단위 권장(1/1~12/31 프리셋 제공) |
| periodEnd | date | Y | periodStart 이후 |
| propertyIds | string[] | N | 미지정 시 전체 물건 포함 |

### 처리 로직
1. LL-020: 기간 선택은 "올해", "작년" 프리셋 버튼 우선 제공(직접 날짜 선택은 보조 옵션)
2. 서버: 기간 내 세대별 월별 납부 확정액(PAID 상태) 합산, 미납액·연체료 별도 집계
3. 자료 상단에 "본 자료는 참고용이며 정식 세무 신고 서식이 아닙니다. 최종 신고 전 세무 전문가 확인을 권장합니다" 고지문 고정 표기
4. PDF 및 엑셀(CSV) 두 형식으로 다운로드 제공
5. LL-019 미납자 명단 대장도 동일한 내보내기 방식(PDF/CSV) 제공

### 예외 처리
| 예외 상황 | 에러 코드 | 사용자 메시지 | UI 처리 |
|-----------|-----------|---------------|---------|
| 해당 기간 납부 데이터 없음 | RES_001 | "해당 기간에 생성된 납부 데이터가 없습니다." | 안내 문구, 다른 기간 재선택 유도 |
| 생성 처리 지연(대량 데이터) | - | "자료 생성 중입니다. 완료 시 알림으로 알려드립니다." | 백그라운드 처리 + 알림 발송 |
| 다운로드 실패 | SERVER_001 | "다운로드에 실패했습니다. 다시 시도해주세요." | 재시도 버튼 |

### API 엔드포인트
```
POST /api/v1/rent/tax-report
Request:  { "periodStart": "2026-01-01", "periodEnd": "2026-12-31" }
Response: { "success": true, "data": { "reportId": "rpt_1", "status": "processing" } }

GET /api/v1/rent/tax-report/{reportId}/download?format=pdf
```

### Use Case 시나리오
**정상**: "작년" 프리셋 선택 → 생성 → 완료 알림 → PDF 다운로드
**예외**: 신규 임대인이라 데이터 없음 → "납부 데이터가 없습니다" 안내 → 안내 문구로 종료
